# Agentic Engine — Roadmap (2026-05-16)

Next 5 iterations on the engine, sequenced. Each item gets its own spec + plan + implementation cycle when we pick it up. This doc is the meta-sequence and rationale; it is NOT a per-item spec.

**Optimization axis:** make daily UX feel right → unlock new agentic behaviors → demonstrate end-to-end on a real workflow. Brain investment (people entities, conversation distillation, etc.) explicitly deferred to a later cycle.

**Ownership:** single-owner across engine + Convex + `app/src/*`. Coordinate with the auth/UX agent's branch on shared files; merge conflicts on `engine/src/`, `convex/agentic/`, and shared schema additions will fall to this side.

**Source of priorities:** `docs/agentic-engine/2026-05-16-corpus-eval.md` — 35-conversation evaluation that surfaced these candidates. Read that first if you're picking up this roadmap mid-stream.

---

## Sequencing principle

Each item ships standalone. Later items use earlier items where dependencies exist, but no item blocks itself on something unbuilt.

- Items 1 and 3 are pure surface + small plumbing. Independent of everything else; can ship in any order if priorities shift.
- Item 2 (auto-chain) is the structural pivot. Several UX cleanups (receipt/follow-up split, status semantics) are bundled with it because they cannot be cleanly built without the underlying behavior change.
- Item 4 (defer) is the heaviest. Builds on item 3's mutation pattern and uses Convex's scheduler primitive.
- Item 5 is the integration end-state. Demonstrates that 1–4 actually compose into a daily-use workflow.

## Universal design principle: primitives vs. UI surfaces

Surfaced explicitly here because it applies across every item: **engine primitives should be universal; UI surfaces should be contextual**. We don't add a permanent button for every primitive — that dilutes the proposal/clarification surface. Examples in this roadmap:

- **Skip** (item 3): the engine primitive exists for every entity; the UI does NOT show a skip button on every proposal. The agent surfaces skip-shaped options contextually; free-text "skip this" works.
- **Defer** (item 4): same pattern. Defer primitive available everywhere; agent decides per-proposal whether to suggest a defer; defer date picker only shows when the agent surfaces a defer option or the user requests one in free text.
- **Auto-chain** (item 2): behavior, not surface. Engine runs it implicitly after EXECUTE; the user never sees a "chain" button.

Future iterations should preserve this discipline. A primitive can ship without a permanent UI affordance and still be valuable — the agent uses it, free text reaches it, and the proposal/clarification surface stays clean.

## Item 1 — Markdown rendering + "Ask X" → clarification fix

**Size:** small. **Domain:** UX (`app/src/components/agent/*`) + engine system prompt.

### Scope

- Wire `react-markdown` + `remark-gfm` into the agent message bubble components. Render backticks as inline code, code fences as monospace blocks, lists/links/tables. Test against existing proposals in the corpus — many already use markdown syntax that isn't being rendered.
- Tighten `DEFAULT_SYSTEM` in `engine/src/runner/claudeSdkRunner.ts`: when the agent needs context only the user has, use `kind: "clarification"` with a populated `question` field, NOT a `proposal` option labeled "Ask X" when X is the user. Add explicit examples of both correct and incorrect shapes.
- UX side: render `kind: "clarification"` distinctly — the `question` field as a prominent prompt, a text input as the primary affordance, plus any suggested-answer options below (still buttons, but framed as "common answers" rather than the only path). Today this kind exists in the schema but is rendered identically to `proposal`.

### Why this first

It's pure surface, ships fast, and fixes two things you see daily. Independent of every other item.

### Acceptance

- All seven "Ask X" anti-pattern proposals from the corpus eval, if re-run today, emit `kind: "clarification"` with a `question`.
- Backticks render as inline code in the drawer.
- A user can type a freeform answer to a clarification question via the rendered text input and it gets sent as the next message.

## Item 2 — Auto-chain after EXECUTE + receipt/follow-up split + status semantics

**Size:** medium. **Domain:** engine (`engine/src/routes/run.ts`, `claudeSdkRunner.ts`) + UX.

### Scope

- **Engine:** after a terminal `execution_result` event lands, automatically enqueue one more discovery turn for the same `entity_ref` with an implicit system-supplied message: *"You just executed `<option_id>`. The entity state has changed. Is this entity truly done, or are there meaningful next moves now? If done, emit `kind: "blocked"` with summary 'task is complete' and empty options. If next moves exist, emit `kind: "proposal"` or `kind: "clarification"` as appropriate."*
- **Engine:** new status semantics — `execution_result` no longer terminal-with-`awaiting_decision`. Instead:
  - If auto-chain produces a `proposal`/`clarification` → status `awaiting_decision` for the follow-up turn.
  - If auto-chain produces a `blocked` with empty options ("truly done") → status `idle`.
  - If auto-chain errors → status `error`.
- **Engine:** auto-chain bounded to one hop per EXECUTE (no runaway loops). If the follow-up auto-chains another EXECUTE later, that's a separate user-initiated cycle.
- **UX:** `execution_result` row renders as a receipt-only success card. No composer prompt, no decide UI. Just the action receipt + tool calls.
- **UX:** follow-up row (proposal or clarification) renders normally with the existing decide UI below the receipt.
- **UX:** when status is `idle` (truly done), no composer prompt active; subtle "Ask anything" affordance available if the user wants to keep going manually.

### Why second

Several Tier-0 UX bugs (receipt/follow-up conflated in one green card, decide UI showing post-EXECUTE, "trudge forward" missing) all stem from the same architectural gap. Splitting them into separate items would either ship broken individually or require throwaway scaffolding. Auto-chain is the pivot that makes the UX cleanups trivially correct.

### Acceptance

- Move-to-medical or move-to-funding actions produce a receipt row + a separate follow-up proposal row asking "anything else for this task?"
- Closing a task ("complete this") produces a receipt row only; status goes `idle`; no decide UI persists.
- The watty→matt-spencer cascade pattern works without the user clicking buttons — auto-chain proposes "also move matt spencer" as a follow-up clarification after the initial move.

## Item 3 — Skip primitive

**Size:** small. **Domain:** Convex schema/mutation + minimal engine wiring.

### Scope

- New `agenticRuns` field: optional `skipped_at: number | null`. Indexed alongside `status` for queue queries.
- New internal mutation `agentic/mutations/skipRun({entity_ref})`. Sets status `idle`, records `skipped_at: Date.now()`, no run enqueued.
- Engine recognizes `EXECUTE: skip` (or equivalent) as a special message: calls `skipRun` and emits a terminal `execution_result` with summary "Skipped — no action taken."
- Agent surfaces skip-shaped options contextually (today's behavior — `close_revisit_later`, `leave_as_is`, etc — already does this; system prompt may need a nudge to make it more frequent for low-confidence proposals).
- Free-text "skip this" / "leave it for now" → agent interprets and executes the skip primitive.
- Burndown queue view (item 5) filters out entities where `skipped_at != null`.
- **No permanent "Skip" button on every proposal.** UI shows skip-shaped options only when the agent includes them in the proposal.

### Why third

Lightweight, immediately useful (24 of 35 conversations in the corpus are sitting in `awaiting_decision` with no obvious next move — skip clears the perpetual queue). Independent of items 1 and 2. The primitive becomes a prerequisite for the burndown queue in item 5 — without skip, the queue accumulates noise.

### Acceptance

- Sending free-text "skip" on an awaiting_decision entity flips status to `idle` and records `skipped_at`.
- A skipped entity doesn't reappear in the burndown queue (item 5 spec will validate this).
- The agent can include a skip option in a proposal — and clicking it skips without a tool roundtrip.

## Item 4 — Defer + scheduled-reopen

**Size:** medium-large. **Domain:** Convex schema/mutation/scheduler + engine + minimal UX.

### Scope

- New `agenticRuns` field: optional `defer_until: number | null` (epoch ms).
- New option attribute on `ProposalOption`: optional `defer_until: string | null` (ISO date or relative like `"2w"`, `"30d"`, `"2026-08-15"`). Agent populates this on defer-shaped options.
- New internal mutation `agentic/mutations/deferRun({entity_ref, defer_until, reason})`. Sets status `idle`, records `defer_until`, schedules a Convex function via `ctx.scheduler.runAt(defer_until_ms, ...)` to call `agentic/actions/postRun` when the timer fires.
- Scheduled action when timer fires: posts `/run` to the engine with system message *"Deferred check-in (set on `<defer_set_at>` for `<reason>`). The entity may have changed since defer. Re-discover current state and propose."* Engine handles like any other discovery turn.
- Defer can be **agent-initiated** (option has `defer_until` populated; clicking it defers) or **user-initiated** (free text "snooze 2 weeks" → agent calls deferRun on next turn).
- UX: deferred entities show a "deferred until `<date>`" badge in the thread; entity drops out of burndown queue until the timer fires.
- UX: a future "scheduled queue" surface lists upcoming auto-reopens with the ability to revoke a defer. Not required for v1; just the data model and primitive.
- **No permanent "Defer" button on every proposal.** Defer date picker only appears when the agent surfaces a defer option or the user explicitly requests one in free text.

### Why fourth

Heaviest item; should go after the lighter ones have landed so we're not committing to it without learning. Builds on item 3's pattern of "primitive without a permanent button" so the discipline is established. The Convex scheduler is the genuinely new piece — relies on a Convex feature we haven't used in this codebase yet.

### Acceptance

- Agent can propose a `defer_until: "2w"` option; EXECUTE on it schedules a re-trigger 2 weeks out.
- After defer fires, a new discovery turn lands with the implicit "deferred check-in" message and produces a fresh proposal grounded in current entity state.
- Revoking a defer (cancel the scheduled function) before the timer fires clears `defer_until` and removes the entity from the upcoming-reopen list.

## Item 5 — End-to-end "inbox to zero" workflow

**Size:** medium. **Domain:** UX (new route) + minimal engine queries.

### Scope

- New route `/agent/inbox` (or similar). Renders a burndown queue of all `todoist:task:*` entries where `project_id === <Inbox>`, sorted by triage value (age, lack of project/labels/due, etc.).
- Queue filters out entities with `skipped_at != null` (item 3) and entities with `defer_until > now` (item 4).
- Keyboard-driven nav:
  - `j` / `k`: move down/up the queue (or `↓`/`↑`).
  - `Enter`: open entity drawer with auto-discovery firing on entry.
  - `1` / `2` / `3` etc.: pick the Nth option from the current proposal.
  - `s`: skip (sends "skip" message, primitive from item 3).
  - `d`: open quick defer picker (item 4) — `d 2w`, `d 30d`, `d tomorrow`.
  - `e`: execute the recommended option.
  - `esc`: close drawer, return to queue.
- Drawer auto-trigger fires discovery on entity entry; auto-chain (item 2) drives follow-ups; user walks the queue executing, skipping, or deferring.
- Goal: clear Inbox to zero (or to whatever subset is genuinely actionable) in one keyboard-driven sitting.

### Why fifth

It's the integration test for the whole roadmap. Each prior item is independently useful, but the workflow only feels right when they compose. If we ship 1–4 and the workflow doesn't feel right, we know which item to revisit. If we shipped 5 first, we'd be building UI atop a behavior layer that doesn't yet support keyboard-driven triage.

### Acceptance

- Cold-open `/agent/inbox`, hit `j` a few times to walk recent tasks, hit `s` to skip a few, hit `d 2w` to defer one, hit `e` to execute the recommended option on a clear one — all without touching the mouse.
- The queue updates reactively as actions land (Convex subscriptions).
- A skipped or deferred entity stays out of the queue until something changes.
- Clear "inbox to zero" run lands in single-digit minutes for a 50-task inbox after some practice.

## Out of scope for this roadmap (deferred to a later cycle)

- **Brain investment** (people entities, conversation distillation, workspace coverage). Highest long-term leverage but explicitly deferred to keep this cycle focused on engine behavior + workflow. See corpus eval's "Brain" section for the specific gaps.
- **Multi-trigger taxonomy** (deadline approach, priority bump, label change, new comment, scheduled sweep). Item 4's scheduler primitive opens the door but the trigger sources themselves are net-new wiring per-entity-type. Probably the next roadmap after this one.
- **Cross-entity reasoning** (compare related tasks, surface duplicates, route batches). Composition of existing primitives; nothing structurally blocks it but no spec yet.
- **Other entity types** (`gmail:thread:*`, `imessage:thread:*`, `calendar:event:*`). The engine is entity-agnostic by design but only Todoist sources are wired. Each new source = one EntitySource implementation + per-type behavior tuning + relevant skill access.

## Process notes for the agent picking this up

- **One spec at a time.** Don't write multiple specs ahead of time. Pick item 1, brainstorm/spec/plan/build/review, then pick item 2.
- **Re-read the corpus eval before each item.** The "why this matters" for each iteration is grounded in specific corpus instances. If those instances feel different by the time you pick up the item, surface the divergence before committing.
- **Run `_adminDigest` periodically.** New corpus data accumulates as items land. A fresh digest before picking up an item should confirm the underlying motivations still hold.
- **Preserve the primitives/surfaces discipline.** Anything that lands as "and also a button for X" gets a second look.
